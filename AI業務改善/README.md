## 一番大きいボトルネック

**「アプリの外側」の作業が手つかず。** 見積アプリ本体は既に非常に良くできています（3モード・共有・内蔵ガイドまで完備）。時間を食っているのは、その前後にある↓の3つ（推測。ただしgit履歴から価格改定対応が今まさに進行中なのは事実）：

1. 依頼文（メール・FAX・電話メモ）→ アプリ入力への**変換作業**
2. 価格改定にともなう**説明文・案内文の執筆**
3. AI開発環境の前提が口頭伝承で、**セッションごとに品質がブレる**こと

## 最優先で直すこと（技術リスク）

**開発ソースがGit管理外**です。`build.js` はリポジトリの外の `../index.html` を参照しており、リポジトリ内の `見積書作成.txt` は3週間前の旧版（価格改定モードなし）。手元PCが壊れると現行ソースを失います。→ 対処案は `06_AI環境監査レポート.md` の承認事項3番。

## 明日からの運用手順（3行）

1. 見積の依頼が来たら、`03` のプロンプトに依頼文を貼って表にしてからアプリに入力する
2. 印刷・送信の前に `04` のチェックリストを30秒なぞる
3. 価格改定の案内は `05` の穴埋め5か所＋数字照合だけで出す

## QA結果（STEP 5）

| 観点 | 結果 |
|---|---|
| 入力不足で止まらないか | 資産03は「書かれていなければ空欄」＋「確認が必要な点」を必ず出す設計。空欄のまま創作しない |
| 出力形式が毎回安定するか | 資産03・05は列名・項目名・順番を固定で指示。**ただしClaude以外の標準モデルでの再現性は未検証**（各ファイルに明記済み） |
| 誤送信・ファイル破壊・機密流出 | テンプレに実在の顧客名・単価は不使用。既存ファイルは一切変更していない（監査の書き直し案も別フォルダ）。機密については監査で公開リポジトリ上の営業所情報・Firestoreルールを**要確認**として指摘 |
| 人間が確認すべき箇所 | 全資産に明記：単価・数量の照合（03）、送信前チェック（04）、金額と適用日の照合（05）、承認3点（06） |
| 次回も再利用できるか | すべてMarkdownでリポジトリにコミット済み。プロンプトは「ここからコピー」ブロックで切り出し可能 |
